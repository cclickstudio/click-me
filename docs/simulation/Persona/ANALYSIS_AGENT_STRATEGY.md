# ClickMe 시뮬레이터 — 분석 에이전트(Debate) & 개선 권고 RAG 전략

> **이 문서의 목적**
> 코딩 에이전트(Claude Code / Codex)가 ClickMe 광고 시뮬레이터의 **분석 레이어**(보고서 §4 크리에이티브 진단 + §5 개선 권고 생성)를 구현할 때 필요한 전략적 컨텍스트를 전달한다.
> 이 문서는 *무엇을, 왜, 어떤 순서로* 만들어야 하는지를 규정한다. 구체적 알고리즘·라이브러리·코드는 구현 단계에서 결정한다.
> **이 문서에 적힌 설계 원칙(특히 §2의 핵심 원칙과 §8의 금지 사항)을 어기는 구현은 거부할 것.**
>
> **선행 문서**: PERSONA_GENERATION_STRATEGY.md (페르소나 생성), SIMULATOR_REPORT_TEMPLATE.md (보고서 출력 양식)
> 이 문서는 두 문서 사이의 빈 칸 — "페르소나 반응 집계 → 보고서 진단·권고 문장"을 채운다.
>
> **v1.1 (2026-06-11) 변경:** ① §3.5 광고 입력·해석 모듈 신설(입력 3종 + 의도 교차검증 + 루브릭 평가 패스) ② §6-5 업종 벤치마크 DB 필수 승격 + 부재 시 강등 규칙 ③ 로스터의 "(6-A)/(6-B)" 참조를 REPORT_TEMPLATE §6-A/§6-B로 명시 ④ 원칙 3 적용 범위 명확화 ⑤ 구현 우선순위·금지 사항 갱신.

## 0. TL;DR (한 문단 요약)

광고 분석은 단일 LLM이 아니라 **4 토론자 + 1 판정자의 멀티 에이전트 토론(debate)**으로 수행한다. 단, 토론의 다양성은 역할 프롬프트가 아니라 **주입 데이터의 비대칭**으로 강제한다 — 각 에이전트는 서로 다른 데이터만 보고, 자기 데이터로만 주장할 수 있으며, 모든 주장에 데이터 ID 인용이 의무다(미인용 주장은 판정자가 기각). 개선 권고는 **진단 근거(내부 페르소나 집계) × 처방 근거(외부 베스트 프랙티스 RAG)의 이중 근거 구조**를 강제하며, 처방 RAG는 플랫폼 공식 가이드(T1)부터 우선 구축한다. 토론은 보고서 §4·§5 생성에만 적용하고, §2 반응 집계는 절대 LLM/토론의 대상이 아니다(코드 집계 영역).

## 1. 배경 및 문제 정의

### 1-1. 우리가 피하려는 실패 모드 2가지

**실패 모드 A — 가짜 토론 (분석 레이어의 동질화)**

에이전트 N개에 역할 프롬프트만 다르게 주는 방식(`"너는 비판적인 분석가야"`)은 다음 문제를 가진다.

- 같은 모델이 같은 데이터를 보고 말투만 바꿔 혼잣말한다. 결론이 단일 LLM과 사실상 동일.
- 이는 PERSONA_GENERATION_STRATEGY §1-1이 경계한 **페르소나 동질화와 동일한 병이 분석 레이어에서 재발**하는 것이다.
- 페르소나 동질화의 해법(다양성을 데이터로 강제)이 여기서도 그대로 해법이다.

**실패 모드 B — 근거 없는 권고 (개선점 할루시네이션)**

단일 LLM에게 "개선점을 제안하라"고 하면:

- 처방이 학습 데이터 평균의 "그럴듯한 일반론"("CTA를 명확히 하세요")으로 수렴한다.
- 근거를 물으면 출처 없는 주장이거나 지어낸 사례가 나온다.
- 마케터가 "그 근거가 뭐냐"고 묻는 순간(반드시 묻는다) 제품 신뢰가 무너진다.

### 1-2. 채택하는 방향

| 실패 모드 | 해법 | 한 줄 원리 |
|---|---|---|
| A. 가짜 토론 | **데이터 비대칭 토론** | 다양성은 역할이 아니라 각자 다른 주입 데이터가 강제한다 |
| B. 근거 없는 권고 | **이중 근거 RAG** | 진단(내부 데이터)과 처방(외부 RAG)을 모두 인용해야 권고 출력 허용 |

### 1-3. 토론 레이어의 위치 — 용어 혼동 주의

본 프로젝트에는 "토론(debate)"이 **두 곳**에 존재한다. 구현·커뮤니케이션에서 반드시 구분할 것.

| | 페르소나 토론 (기존) | **분석가 토론 (이 문서)** |
|---|---|---|
| 출처 | PERSONA_STRATEGY §4 검문소 2 (Deepsona Debate Agent) | 본 문서 |
| 참여자 | 가상 소비자 페르소나들 | 전문가 시점 분석 에이전트들 |
| 목적 | 응답 분산 확장 (쏠림 방지) | 진단·권고의 품질과 근거 강화 |
| 파이프라인 위치 | 페르소나 응답 생성 직후 (집계 전) | 집계 완료 후 (보고서 §4·§5 생성) |
| 우선순위 | 여유 시 (필수 아님) | **필수** (이 문서의 본체) |

## 2. 핵심 설계 원칙 (절대 위반 금지)

1. **토론의 다양성은 데이터 비대칭이 강제한다.** 각 에이전트는 자신에게 배정된 데이터만 본다. 전체 데이터를 모든 에이전트에게 주는 구현은 토론을 가짜로 만든다.
2. **모든 주장에 데이터 ID 인용 의무.** 인용 없는 주장은 판정자가 기각한다. 이것이 할루시네이션의 구조적 차단 장치다.
3. **§2 반응 집계는 토론·LLM의 대상이 아니다.** 숫자는 집계 엔진(코드)이 만들고, 토론은 그 숫자의 *해석*만 다룬다. (PERSONA_STRATEGY 원칙 1 "속성 생성에 LLM 금지"의 분석 레이어 버전) **적용 범위 주의:** 이 원칙은 §2 집계 수치에 한정된다. §4 차원 점수는 루브릭 기반 LLM 평가 패스(§3.5)가 토론 **전**에 1회 산출하며, 토론은 그 점수를 수정하지 않고 해석·진단 문장만 다룬다.
4. **개선 권고는 이중 근거를 강제한다.** 진단 근거(내부) 또는 처방 근거(외부 RAG) 중 하나라도 없으면 정식 권고로 출력하지 않는다.
5. **불일치는 숨기지 않는다.** 에이전트 간 합의되지 않은 쟁점은 "분석가 간 이견"으로 보고서에 정직하게 노출한다. 억지 합의는 가짜 숫자와 같은 죄다.
6. **검색 실패는 빈 채로.** 처방 RAG가 근거를 못 찾으면 해당 권고를 "일반 권고(외부 근거 없음)"로 강등하거나 출력 보류한다. (보고서 양식 "빈 데이터는 빈 채로" 원칙)

## 3. 분석 파이프라인 — 전체 흐름

```
[입력]                      [토론 — 이 문서의 본체]              [출력]
페르소나 응답 집계 완료      Round 1   Round 2   Round 3
(§2 JSON, 코드 생성)    →   입장 발표 → 교차 반박 → 판정      →   §4 진단 카드
광고 원본                   (병렬 4콜)  (병렬 4콜)  (1콜)          §5 개선 권고
벤치마크 DB                                                      (+ 이견 표시)
처방 RAG                         ↑
                          데이터 비대칭 주입
```

- 합계 **9 LLM 콜** (4+4+1). 페르소나 시뮬레이션 ~1,000콜 대비 오버헤드 약 1% — 비용 우려 없음.
- Round 1·2는 병렬 실행 (LangGraph parallel node). Round 간에는 순차 의존.

## 3.5. 선행 단계 — 광고 입력·해석 모듈 (신설)

토론과 페르소나 시뮬레이션 **모두에 선행**하는 입력 처리 단계. (기존 문서들이 전제하면서도 규정하지 않았던 빈칸을 여기서 채운다.)

### 3.5-1. 사용자 입력 3종

| 입력 | 필수 여부 | 내용 | 흘러가는 곳 |
|---|---|---|---|
| **광고 소재** | 필수 | 이미지 / 영상 / 카피 | 해석 모듈 → AGT-CD 주입 + 루브릭 평가 패스 |
| **광고 세부 정보** | 필수 | 업종 카테고리, 상품/서비스 분류, 광고 목적 | ① REPORT §0 헤더 ② §6-A 벤치마크 매칭 키 + 처방 RAG `industry` 필터 ③ §1 종합 점수의 목적별 가중치 ④ **선언 의도** — §3-1 의도-반응 비교의 기준점 |
| **타깃 설정** | 선택 | 연령대 · 성별 | PERSONA_STRATEGY 단계 1의 `target_filter`. **기본 = 미지정(전체 인구 샘플링)** |

### 3.5-2. 해석 모듈의 출력 4종

1. **구조화 광고 분석** — 훅·카피·CTA·비주얼 요소 (멀티모달 LLM 1패스)
2. **감지 의도** — 광고가 *실제로* 말하고 있는 카테고리·타깃·메시지
3. **의도 교차검증** — 사용자 **선언 의도** vs **감지 의도** 비교. 사용자 입력이 우선하되 자동 분류를 끄지 않는다 — 충돌 시 경고를 표시하며, **그 불일치 자체가 §4 "타깃 적합성" 진단의 1급 근거**가 된다. (예: 선언 "30대 건강음료" vs 감지 "20대 감성 에너지드링크 코드")
4. **차원별 루브릭 평가 패스** — §4 진단 항목(공통 7 + 한국 특화 3)의 점수를 **토론 전에 1회 산출.** 점수 + 산출 근거가 AGT-CD 주입 데이터에 포함되며, 토론은 점수의 *해석*과 진단 문장 생성을 담당한다 (원칙 3의 적용 범위 참조).

> **일정:** 입력 파싱 + 구조화 분석 + 교차검증 + 루브릭 패스 합산 ~1.5일. 구현 우선순위 §7에 필수 항목으로 반영됨.

## 4. 에이전트 명세 — 4 토론자 + 1 판정자

### 4-1. 에이전트 로스터

| ID | 에이전트 | 시점(페르소나) | 주입 데이터 (이것**만** 본다) | 토론 임무 |
|---|---|---|---|---|
| `AGT-CD` | 크리에이티브 디렉터 | 15년차 CD. "광고 그 자체"를 본다 | ① 광고 해석 모듈 출력(§3.5 — 구조화 분석 + 루브릭 평가 패스 점수·근거) ② §4 진단 차원 루브릭 ③ 플랫폼 크리에이티브 가이드 RAG (T1) | 훅·구조·표현·CTA 관점의 주장. **소비자 반응 데이터를 보지 못하므로** 순수 제작 관점이 유지된다 |
| `AGT-DA` | 소비자 데이터 분석가 | 리서치 펌 퀀트. "페르소나가 한 말"만 믿는다 | ① §2 집계 JSON 전체 (AISAS 퍼널·구매의도 분포·거부 사유·감정·세그먼트) ② 대표 발화 풀 | 페르소나 반응의 대변인. CD의 주관적 주장을 "데이터에 그런 근거 없음"으로 견제 |
| `AGT-MS` | 시장 전략가 | 한국 시장 미디어 플래너 | ① 업종 벤치마크 DB (§6-5 — MVP 필수) ② 트렌드 RAG (REPORT §6-B — 여유 시) ③ 매체 환경 데이터 (여유 시) | "업계 기준에서 이 수치는 정상/이상" 맥락 부여. 내부 데이터의 과대해석 견제 |
| `AGT-SK` | 회의론자 (Devil's Advocate) | 광고주 측 깐깐한 검수자 | ① 거부 사유 집계 ② 최저점 세그먼트 데이터 ③ 응답 분산 경고 플래그 ④ **타 에이전트 발언록 (Round 2부터)** | 모든 긍정 결론에 반례 요구. "그 근거로 그 결론이 나오는가?" 논리 비약 공격 전담 |
| `AGT-JD` | 판정자 (Judge) | 합의 도출자 | **새 데이터 없음** — Round 1·2 토론 로그만 | 인용 없는 주장 기각 → 합의/이견 정리 → §4 진단문·§5 권고 초안 생성 |

### 4-2. 데이터 비대칭 매트릭스 (구현 시 접근 제어표)

```
                    AGT-CD   AGT-DA   AGT-MS   AGT-SK   AGT-JD
광고 원본             ✅       ❌       ❌       ❌       ❌
진단 차원 루브릭       ✅       ❌       ❌       ❌       ❌
§2 집계 JSON          ❌       ✅       ❌       부분*     ❌
대표 발화 풀          ❌       ✅       ❌       ❌       ❌
벤치마크 DB           ❌       ❌       ✅       ❌       ❌
트렌드 RAG (§6-B)     ❌       ❌       ✅       ❌       ❌
처방 RAG (T1~T4)      ✅       ❌       ❌       ❌       ❌
거부·최저점·분산경고   ❌       ❌       ❌       ✅       ❌
토론 로그             ❌       ❌       ❌       R2부터    ✅

* AGT-SK의 "부분" = 부정 신호만 (거부 사유, 최저 세그먼트, 분산 경고).
  긍정 데이터를 주지 않아 비판자 역할이 구조적으로 유지된다.
```

> **구현 규칙:** 이 매트릭스는 프롬프트 조립 코드에서 하드코딩으로 강제한다.
> "편의상 전부 다 넣자"는 리팩토링 제안은 §2 원칙 1 위반 — 거부할 것.

### 4-3. 발언 스키마 (모든 에이전트 출력의 구조화 형식)

```json
{
  "agent_id": "AGT-DA",
  "round": 1,
  "claims": [
    {
      "claim_id": "DA-R1-03",
      "statement": "CTA 불명확이 Action 이탈의 주원인이다",
      "evidence_refs": ["AGG.rejection.unclear_action=0.31", "AGG.funnel.search_to_action_drop=0.15"],
      "confidence": "high",
      "targets_claim": null
    }
  ]
}
```

| 필드 | 규칙 |
|---|---|
| `evidence_refs` | **빈 배열 금지.** 집계 JSON 경로, RAG 청크 ID, 벤치마크 키 중 하나 이상 |
| `targets_claim` | Round 2 반박 시 공격 대상 claim_id 명시 (없으면 신규 주장) |
| `confidence` | high / medium / low — Judge의 가중 판단 입력 |

## 5. 토론 프로토콜 — 3 라운드

### Round 1 — 입장 발표 (병렬 4콜)
- 각 에이전트가 자기 데이터만으로 독립 분석. 타 에이전트 발언 비공개 (오염 방지).
- 출력: 발언 스키마 준수 claims 3~7개.

### Round 2 — 교차 반박 (병렬 4콜)
- Round 1 전체 발언록 공개. 각 에이전트는 **자기 데이터와 모순되는 타인 주장만** 공격 (`targets_claim` 필수).
- 동의는 발언하지 않는다 (토큰 절약 + 반박 신호의 밀도 유지).
- AGT-SK는 이 라운드에서 논리 비약 공격을 전담한다 — 데이터 모순이 아니어도 "근거→결론 비약"이면 공격 가능.

### Round 3 — 판정 (1콜)
AGT-JD의 처리 순서:

```
1. 기각   : evidence_refs 없는/유효하지 않은 claim 제거 (기각 사유 로그)
2. 분류   : 생존 claim을 [합의 / 이견 / 단독주장]으로 분류
            합의 = 2개 이상 에이전트의 데이터가 같은 방향
            이견 = 데이터 간 충돌 (예: CD "훅 강함" vs DA "Attention 평균 이하")
3. 생성   : 합의 → §4 진단 카드 문장 + §5 권고 초안
            이견 → 보고서에 "분석가 간 이견" 블록으로 정직 노출
4. 검증   : 생성 문장의 모든 수치·인용이 원본 evidence_refs와 일치하는지 후처리 검사
            (불일치 시 해당 문장 폐기 — LLM 생성문의 최종 QA)
```

> **이견 노출의 가치:** 실제 리서치 펌 보고서도 분석가 의견 차이를 표기한다.
> "AI가 만장일치로 단언"보다 "관점에 따라 평가가 갈림"이 오히려 신뢰를 산다.
> 기획서 핵심 전제("직감보다 나은 근거", 단언 아님)와 일치.

## 6. 개선 권고 — 이중 근거 구조

### 6-1. 권고의 성립 조건

```
개선 권고 = 진단 근거 × 처방 근거

진단 근거 (내부)  : "무엇이 망가졌나"  ← §2 페르소나 집계 (토론에서 확정된 claim)
처방 근거 (외부)  : "무엇이 통하나"    ← 베스트 프랙티스 RAG (아래 6-3)

둘 다 있음        → 정식 권고 (§5 상단, 영향도 순)
진단만 있음       → "일반 권고 (외부 근거 없음)" 라벨로 강등 표시
처방만 있음       → 출력 금지 (문제가 확인 안 된 처방은 영업 멘트일 뿐)
```

### 6-2. 권고 객체 스키마 (§5 출력 + 광고 생성팀 4-3 입력 인터페이스)

```json
{
  "recommendation_id": "REC-001",
  "priority": 1,
  "severity": "critical",
  "target_dimension": "cta_clarity",
  "current_state": "영상 종료 시 행동 지시 부재 — '지금 만나보세요'는 행동 불명확",
  "proposal": "단일 행동 지시로 교체 (예: '첫 구매 30% 쿠폰 받기')",
  "diagnosis_evidence": ["AGG.rejection.unclear_action=0.31", "CLAIM:DA-R1-03"],
  "prescription_evidence": ["RAG:youtube-abcd-direct-007", "RAG:kakao-guide-cta-012"],
  "expected_impact": "high",
  "impact_note": "Action 단계 업종 평균 수준 회복 기대"
}
```

| 필드 | 규칙 |
|---|---|
| `expected_impact` | high / medium / conditional 3단계 정성. **"개선 시 +12점" 같은 정밀 수치 약속 금지** |
| `prescription_evidence` | RAG 청크 ID. 빈 배열이면 자동으로 "일반 권고" 강등 |
| 전체 객체 | 이 스키마가 그대로 **광고 생성 기능(기획서 4-3)의 입력**이 된다 — 팀 간 인터페이스 계약. 변경 시 광고 생성팀 합의 필수 |

### 6-3. 처방 RAG — 데이터 소스 4계층

| 계층 | 소스 | 역할 | 수집 난이도 | 구현 시점 |
|---|---|---|---|---|
| **T1** | 플랫폼 공식 크리에이티브 가이드: **YouTube ABCD** (Attract·Brand·Connect·Direct), **Meta Performance 5** / 크리에이티브 가이드, TikTok Creative Codes, **네이버·카카오모먼트 광고 소재 가이드** | 처방의 주력. 플랫폼이 자사 대규모 데이터로 검증한 권고 (ABCD는 외부 검증 연구 존재) | 낮음 (공개 문서) | **필수 — 최우선** |
| **T2** | 한국 공공 연구: **KOBACO ADSTAT**(방송통신광고통계시스템) 연구보고서·통계 DB, KISDI·언론진흥재단 광고 관련 보고서 | 한국 특화 처방의 학술 근거. 경쟁사가 못 흉내내는 해자 (페르소나 전략과 동일 논리) | 낮음~중간 (공공 공개) | 여유 시 |
| **T3** | 효과 입증 사례: **에피 어워드 코리아(Effie Korea)** 수상작 — *성과 기준* 시상이므로 "성과 낸 광고"의 사례집 | 사례 기반 처방. "비슷한 문제를 이렇게 풀어 성과 냈다" | 중간 (케이스 요약 필요) | 여유 시 |
| **T4** | 학술 문헌: 광고학연구·광고연구 등 한국 광고학 저널, 설득 커뮤니케이션 문헌 | 진단 차원별 이론 근거 (발표 인용용) | 중간 | 발표 준비 시 핵심 10~20편 |

> **7/8 일정 판단: T1만으로 데모 성립.** ABCD + Performance 5 + 네이버/카카오 가이드만 임베딩해도 §4 진단 차원 대부분에 처방이 매핑된다.

### 6-4. 처방 RAG 청크 메타데이터 스키마 (필수)

```json
{
  "chunk_id": "youtube-abcd-direct-007",
  "source": "YouTube ABCD Framework",
  "source_tier": "T1",
  "dimension": ["cta_clarity"],
  "media_type": ["video"],
  "industry": ["all"],
  "language": "ko",
  "retrieved_date": "2026-06-15",
  "content": "..."
}
```

| 필드 | 규칙 |
|---|---|
| `dimension` | **§4 진단 항목(공통 7 + 한국 3)과 1:1 매핑되는 enum.** 자유 텍스트 금지 — 이 태그가 검색 정밀도의 전부다 |
| `media_type` | video / image / text / all — 광고 포맷 필터 |
| `industry` | 업종 enum 또는 "all" |
| `source_tier` | T1~T4 — 권고 출력 시 근거 등급 표시용 |
| 검색 방식 | `dimension` + `media_type` 메타데이터 필터 → 벡터 유사도. 필터 없는 전체 벡터 검색 금지 (정밀도 붕괴) |

### 6-5. 업종 벤치마크 DB — 필수 승격 (REPORT §6-A의 데이터 원천)

업종 평균은 §6-A에만 쓰이는 게 아니다. **REPORT §1 종합 점수("업종 평균 대비 ±") · §2-1 퍼널 오버레이 · §4 전 진단 항목의 업종평균 컬럼**에 들어가는, 보고서 전체의 하중을 받는 기둥이다. 따라서 "여유 시"가 아니라 **필수**다. 단, 자동 수집 파이프라인이 아니라 **손 큐레이션 정적 JSON 1개**로 MVP가 성립한다 (~0.5–1일).

| 항목 | 규칙 |
|---|---|
| 범위 | 업종 10~20개 × 지표 (평균 CTR 레인지, 가능 시 영상 완주율·퍼널 단계 평균) |
| 키 형식 | `BENCH.{industry}.{metric}` — 발언 스키마의 `evidence_refs` 인용 형식과 직결 |
| 행 필수 필드 | 수치 + **출처명 + 기준연도** (출처 없는 행 등록 금지 — REPORT §6-A 규칙과 동일) |
| 출처 후보 | 나스미디어·메조미디어 공개 디지털 광고 리포트, KOBACO 공개 통계, Meta/Google 공식 벤치마크 자료 |
| 갱신 | 정적 파일, 분기 단위 수동 갱신 |

**해당 업종 데이터 부재 시 — 설계된 강등 (침묵 실패 금지):**
- 보고서: §6-A를 "벤치마크 데이터 없음"으로 정직 표기 (기존 규칙).
- 토론: **AGT-MS를 해당 시뮬레이션에서 결석 처리**하고, Judge가 §4에 "시장 맥락 분석 불가(벤치마크 부재)"를 명시 → 3인 토론으로의 우아한 강등을 *설계된 동작*으로 문서화한다.
- ❌ MS에게 인접 업종 수치를 대신 주입하는 것 금지 — 가짜 맥락은 가짜 숫자와 같은 죄다 (원칙 5·6과 동일 논리).

## 7. 구현 우선순위 (마감: 2026-07-08)

**원칙: PERSONA_STRATEGY와 동일 — 어설퍼도 한 바퀴가 끝까지 도는 게 우선.**

### 필수 — 데모 가능 최소선
1. **광고 해석 모듈 (§3.5)** — 입력 3종 파싱 + 구조화 광고 분석 + 의도 교차검증 + 차원별 루브릭 평가 패스. (~1.5일)
2. **T1 처방 RAG 구축** — ABCD·Performance 5·네이버/카카오 가이드 수집 → 메타데이터 스키마로 청킹·임베딩. (~2일)
3. **업종 벤치마크 DB (§6-5)** — 정적 JSON 손 큐레이션 + 부재 시 강등 규칙 구현. (~0.5–1일)
4. **데이터 비대칭 주입 코드** — 4-2 매트릭스를 프롬프트 조립 레이어에 하드코딩. (~1일)
5. **Round 1~3 토론 그래프** — LangGraph 병렬 노드 (4콜) → 병렬 (4콜) → Judge (1콜). 발언 스키마 강제 (structured output). (~2일)
6. **Judge 기각·분류 로직** — evidence_refs 유효성 검사(집계 JSON 경로 + RAG 청크 ID + 벤치마크 키 실존 확인) → 합의/이견 분류. (~1일)
7. **권고 객체 생성** — 이중 근거 검증 → REC 스키마 출력 → 보고서 §5 렌더링. (~1일)
8. **이견 노출 블록** — §4에 "분석가 간 이견" 표시 구현. (~0.5일)

### 여유 되면
- T2 (KOBACO ADSTAT) RAG 추가
- 트렌드 RAG (REPORT §6-B) 구축 — AGT-MS 주입 데이터 ②번 확장 (§4 "트렌드 코드 정합성" 차원은 데이터 부재 시 "—" 처리 가능하므로 MVP 미구현 허용)
- AGT-SK 공격 패턴 고도화 (논리 비약 검출 프롬프트 개선)
- 토론 로그 뷰어 (디버깅 + 발표 시 "토론 과정" 시연용 — 발표 임팩트 큼)

### 발표 후 (Phase 2+)
- T3 (에피어워드 사례) / T4 (학술 문헌) RAG 확장
- Round 2 다회전 (수렴할 때까지 반복) 실험
- Judge 기각률·이견률 모니터링 대시보드 (품질 지표화)

## 8. 금지 사항 (구현 시 하지 말 것)

- ❌ 모든 에이전트에게 **전체 데이터를 주입하기.** (데이터 비대칭 붕괴 = 가짜 토론 — §2 원칙 1 위반)
- ❌ 역할 프롬프트만 다르게 한 채 **동일 데이터로 토론시키기.** (실패 모드 A 그 자체)
- ❌ `evidence_refs` 없는 주장을 **Judge가 통과시키기.** (할루시네이션 차단 장치 무력화)
- ❌ §2 반응 집계 수치를 **토론·LLM이 생성하거나 수정하기.** (숫자는 코드의 영역)
- ❌ 에이전트 간 이견을 **억지로 합의시켜 숨기기.** (가짜 만장일치 — 정직성 원칙 위반)
- ❌ 처방 근거 없는 권고를 **정식 권고로 출력하기.** ("일반 권고" 강등 라벨 필수)
- ❌ 권고에 **"개선 시 +N점" 같은 정밀 수치 효과 약속하기.** (검증 불가 단언)
- ❌ RAG 검색을 **메타데이터 필터 없이 전체 벡터 검색으로 구현하기.** (정밀도 붕괴)
- ❌ 페르소나 토론(검문소 2)과 분석가 토론(이 문서)을 **하나의 모듈로 합치기.** (목적·위치·데이터가 다른 별개 레이어)
- ❌ REC 스키마를 **광고 생성팀 합의 없이 변경하기.** (팀 간 인터페이스 계약)
- ❌ 벤치마크 부재 업종에서 **인접 업종 수치를 대용하거나 평균을 추정 생성하기.** (§6-5 강등 규칙 위반 — MS 결석 처리가 정답)
- ❌ 루브릭 평가 패스가 산출한 §4 차원 점수를 **토론이 수정하기.** (점수는 토론 전 1회 산출 — 토론은 해석·진단 문장만)

## 10. 한 문장 요약

**토론의 다양성은 데이터 비대칭으로 강제하고, 모든 주장은 데이터 ID를 인용해야 살아남으며, 개선 권고는 "내부 진단 × 외부 처방"의 이중 근거가 있어야 출력된다 — 근거 없는 문장은 이 시스템에서 구조적으로 생존할 수 없다.**
