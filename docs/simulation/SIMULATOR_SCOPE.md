# ClickMe 시뮬레이터 — 구현 범위 & 데이터 소유권 (SCOPE)

> 시뮬레이터 팀 담당 = **페르소나 생성 + 반응 집계(숫자 생산)**. 분석·리포트(문장 생산)는 별도 팀.
> 경계는 `PERSONA §3.5 반응 스키마`와 `시뮬레이션집계`를 **인터페이스 계약**으로 고정한다. 분석팀은 이 둘을 **읽기만** 한다.

## 0. 소유권 원칙

- **숫자는 시뮬레이터가, 문장은 분석팀이** (REPORT 핵심 구현 원칙 1).
- 시뮬레이터는 분석/리포트 테이블에 **의존·쓰기하지 않는다.** 핸드오프는 단방향(시뮬레이터 → 분석팀).
- 9개 테이블의 ORM 정의는 공통부 `core/models.py`에 들어가므로, 추가 시 **협업 규칙(DB 단독 변경 금지 + Alembic)** 을 따른다.

## 1. 구현 대상 — 시뮬레이터 소유 9테이블

| 테이블             | 역할                                   | 쓰기 주체                | 비고                                  |
| ------------------ | -------------------------------------- | ------------------------ | ------------------------------------- |
| `프로젝트`         | 캠페인 단위                            | 공통/매니지먼트 (참조)   | 시뮬레이터는 **읽기만**, FK 참조용     |
| `광고`             | 광고 입력(소재·카피·타깃)              | 입력 폼 (참조)           | 시뮬레이터는 **읽기/입력**             |
| `광고해석`         | 광고 이해(vision) 구조화 산출          | **시뮬레이터**           | exposure/페르소나 생성의 입력          |
| `패널`             | 고정 패널 빌드(버전·시드·grounding)    | **시뮬레이터**           | §3.6 고정 패널 운영                    |
| `페르소나`         | 패널 멤버(OCEAN·미디어·소비·서사)      | **시뮬레이터**           | 프로필만 캐시, 반응은 매 실행 생성     |
| `시뮬레이션`       | 실행 메타(상태·샘플수·버전·에러)       | **시뮬레이터**           | 파이프라인 오케스트레이션 단위          |
| `페르소나반응`     | 반응 생성(AISAS·구매의도·거부·발화)    | **시뮬레이터** ★         | **계약** — §3.5 스키마, 분석팀 입력     |
| `루브릭점수`       | 루브릭 평가 패스(차원별 점수, 숫자)    | **시뮬레이터** (경계 주의)| §4 진단 입력. LLM 호출 위치 1회 확정    |
| `시뮬레이션집계`   | 집계 엔진 산출(CTR·CI·구매·거부율)     | **시뮬레이터** ★         | **계약** — 분석팀·리포트 입력          |

★ = 팀 간 인터페이스 계약. 컬럼 변경 시 분석팀·렌더링 코드와 **동시 수정**.

## 2. 범위 밖 — 분석/리포트 (건드리지 않음)

| 테이블                          | 소유      | 시뮬레이터 관계        |
| ------------------------------- | --------- | ---------------------- |
| `토론세션` · `토론발언`         | 분석팀    | 무관 (import·쓰기 금지) |
| `진단` · `개선권고` · `보고서`  | 분석팀    | 무관 (import·쓰기 금지) |
| `조직` · `조직멤버` · `사용자`  | 공통/인증 | FK 참조만               |

- 분석팀은 `페르소나반응` + `시뮬레이션집계`(+ `루브릭점수`)를 **읽어서** 토론·진단·권고·보고서를 만든다.
- 시뮬레이터 코드는 위 분석 테이블/모듈을 **import 하지 않는다.** 교환은 DB 계약 컬럼으로만.

## 3. 파이프라인 — 시뮬레이터 책임 구간

```
광고해석(vision) → 패널/페르소나(생성·캐시) → exposure → deliberation
                 → 반응 구조화(§3.5) → 루브릭 평가(숫자) → 집계 엔진 → [핸드오프]
─────────────────────────── 시뮬레이터 ───────────────────────────┘
                                          토론 · 진단 · 권고 · 보고서 → [분석팀]
```

## 4. 도메인 매핑 (`backend/domain/simulation/`)

| 디렉터리         | 채울 내용                                                            |
| ---------------- | ------------------------------------------------------------------- |
| `contracts/`     | 반응(§3.5)·집계 스키마 포트, enum taxonomy(거부·감정·이탈)           |
| `service/`       | 파이프라인 오케스트레이션 + SSE (현 `simulation_service.py` 정착지)   |
| `repositories/`  | 위 9테이블 CRUD (영속화 — 현재 미구현)                              |
| `model/`         | 시뮬레이션 상태·전이(QUEUED→RUNNING→DONE/FAILED)                     |
| `agent/debate/`  | **분석팀 경계** — 시뮬레이터는 비워두거나 분석팀에 이관             |

## 5. 현 구현 갭 (착수 전 정리 필요)

1. **영속화 없음** — `service`가 `_tasks` in-memory dict만 사용. `repositories/` 비어 있음 → 9테이블 매핑 필요.
2. **스키마 불일치** — `core/schemas.py`(분포형 signals) vs ERD(`aisas` bool·`purchase_intent` INT 1~5·enum 태그) 정합 1회 필요.
3. **고정 패널 미구현** — 현재 매 실행 페르소나 재생성. §3.6 패널 캐시 도입 필요.
4. **집계 엔진 미분리** — 집계가 `service` 인라인. `시뮬레이션집계` 컬럼(`ci_low/high`·`rejection_rate`·`variance_warning`) 산출 로직 부재.
5. **enum taxonomy 미확정** — 거부·감정·이탈 사유는 REPORT §2-3과 **1회 확정 후** 코드 동기화.

## 6. 금지 (경계 보호)

- 분석 테이블(`진단`·`개선권고`·`보고서`·`토론*`) import·쓰기 금지.
- `페르소나반응`·`시뮬레이션집계` 스키마 변경은 분석팀과 동시 합의(계약).
- `core/models.py`·`docs/db-schema.md` 변경은 공지 + Alembic (협업 규칙).
