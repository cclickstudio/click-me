# ClickMe 백엔드 — DDD + 헥사고날 구조 안내 (simulation 도메인 기준)

> 이 폴더(`domain/simulation/`)는 **DDD + 헥사고날(Ports & Adapters)** 아키텍처를 반영해 만든 골격이다.
> **지금은 폴더 구조와 `service` 스텁만 있고, 시뮬레이션 비즈니스 로직은 아직 들어 있지 않다.**
> 이 문서는 "각 폴더가 헥사고날에서 무슨 역할인지"와 "코드를 어디에 넣어야 하는지"를 전달한다.
> 같은 패턴을 **이미 구현해 둔 레퍼런스**는 형제 도메인 `domain/generator/`, `domain/management/`다.

---

## 0. 한눈에 — 왜 이렇게 나눴나

핵심은 **의존성 역전**이다. 도메인 로직(순수 규칙)은 바깥(LLM·DB·외부 API)을 **모른다.** 바깥과 닿는 코드는 전부 가장자리(adapters)로 밀어내고, 도메인은 **포트(인터페이스)** 만 바라본다. 그래서 mock ↔ 실연동 교체가 한 곳(`wiring`)에서 끝난다.

```
        ┌─────────── 바깥 세계 ───────────┐
        │  LLM(Gemini) · DB · 외부 API     │
        └──────────────┬──────────────────┘
                       │ adapters (구현체)
                       ▼
   contracts(Port) ◀── 도메인은 인터페이스만 의존 ──┐
        ▲                                            │
        │ tools(순수 규칙) · graph(흐름)             │
        │                                            │
   service(구동) ─── 결과 조립 · 진행률 · 영속화 지휘 ┘
        ▲
   wiring (Composition Root — 어떤 adapter를 꽂을지 여기서만 결정)
```

- **도메인(`contracts`/`tools`/`graph`)은 안쪽**, **`adapters`/`repositories`는 바깥쪽.**
- 안→밖 직접 호출 금지. 밖은 안의 **포트(`contracts`)를 구현**해서 주입된다.
- 숫자(집계 KPI)는 `tools`가 코드로 계산, LLM은 서사·진단 '문장'만. (도메인 규칙)

## 1. 폴더 = 헥사고날 레이어 매핑

```
domain/simulation/
├── contracts/      🤝 Port + DTO — 도메인이 주고받는 인터페이스/스키마/enum. 외부 의존 0
├── data/           📊 grounding 데이터 — 한국 통계 분포 JSON + 로더
├── tools/          🧩 도메인 코어 — 순수 규칙/계산(샘플링·집계). LLM 거의 없음(service역할이라 보면됨됨)
├── adapters/       🔌 Driven Adapter — Port 구현체(mock · 실 LLM/Gemini)
├── graph/          🔀 오케스트레이션 — LangGraph 파이프라인 흐름 (graph/agent/ 포함)
├── service/        ▶️ Application — 그래프 구동 · 진행률(SSE) · 결과/영속화 지휘(오케스트레이터 역활)
├── repositories/   💾 Persistence Adapter — 테이블 CRUD (SQL은 여기에만)
├── models.py       🗄️ ORM 엔티티 (PostgreSQL/SQLite 호환)
└── wiring.py       ★ Composition Root — adapter를 Port에 꽂는 유일한 지점
```

| 레이어 | 헥사고날 용어 | 한 줄 역할 |
| --- | --- | --- |
| `contracts/` | **Port** (인터페이스) + DTO | "무엇을 주고받나"의 약속. 타 도메인·분석팀과 교환 단위 |
| `tools/` | Domain core | 숫자를 만드는 순수 부품 (외부 모름) |
| `data/` | Domain resource | 페르소나 grounding용 실데이터 |
| `adapters/` | Driven Adapter | Port를 구현해 바깥(LLM)과 연결. mock ↔ 실연동 |
| `repositories/` | Driven Adapter (영속화) | DB 접근 전담. SQL이 새어나가지 않게 격리 |
| `graph/` | Orchestration | 노드 흐름 정의. 노드는 주입받은 부품을 호출 |
| `service/` | Application(Driving) | 그래프를 돌리고 결과 조립·저장 지휘 |
| `wiring.py` | **Composition Root** | mock/실 구현체 선택을 여기서만 결정 |

> 레퍼런스: `domain/management/contracts/platform.py`(Port 정의)와 `domain/management/wiring.py`(`build_reader`/`build_writer`로 mock↔Meta 어댑터 주입)가 이 패턴의 작동 예시다.

## 2. 의존 방향 & 핵심 규칙 (코드 넣을 때 지킬 것)

```
service ──구동──▶ graph ──(노드가 호출)──▶ tools · adapters · repositories
                              ▲ 모든 구현체는 wiring 이 주입
```

1. **단방향 의존** — `service`→`graph`→부품. 부품은 `service`를 모른다. 안쪽(`tools`/`contracts`)이 바깥쪽(`adapters`)을 import 하지 않는다.
2. **wiring.py 가 유일한 조립점** — `MockXxx()`/`GeminiXxx()`를 다른 곳에 직접 박지 않는다. 교체는 여기서만.
3. **계약으로만 교환** — 타 도메인(generator·management)·분석팀과는 `contracts` 스키마로만. 내부 모듈 직접 import 금지.
4. **숫자 vs 문장** — KPI 수치는 `tools`(집계)가 코드로 계산. LLM은 서사·진단 '문장'에만.
5. **SQL은 `repositories`에만** — `service`는 "저장 지휘"만, CRUD/SQL은 repository.

## 3. 현재 구현 상태 (★ 솔직하게)

| 위치 | 상태 |
| --- | --- |
| `contracts/`, `data/`, `tools/`, `adapters/`, `repositories/`, `graph/agent/` | **빈 폴더** — 골격만 생성됨 |
| `graph/__init__.py` | 빈 파일 |
| `models.py`, `wiring.py` | 주석만 (엔티티/조립 로직 미작성) |
| `service/simulation_service.py` | **유일한 실코드.** 단, 아직 헥사고날을 따르지 않음 (아래) |

**`service/simulation_service.py`의 현 한계 (마이그레이션 대상):**
- 실 로직을 도메인 밖 레거시 위치에서 import: `tools/ad_analysis/vision.py`, `tools/persona/factory.py`, `tools/simulation/{exposure,deliberation,ssr_scorer}.py`, 스키마는 `core/schemas.py`.
- `service`가 SQL(`INSERT/UPDATE`)을 **직접 실행** → 규칙 5 위반 (`repositories`로 이전해야 함).
- `graph`(LangGraph)를 거치지 않고 service가 파이프라인을 직접 돌림 → 규칙 1대로 `graph`로 옮겨야 함.

> 즉 **폴더는 헥사고날, 코드는 아직 절차형**이다. 다음 작업은 레거시 `tools/`·`core/schemas.py`의 시뮬레이션 코드를 이 도메인의 해당 레이어로 이전하고, `contracts`에 Port/DTO를, `wiring`에 조립을 채우는 것이다.

## 한 문장

**`domain/simulation/`은 generator·management와 동일한 DDD 헥사고날 골격이다 — 도메인(`contracts`·`tools`·`graph`)은 바깥을 모르고, 바깥(`adapters`·`repositories`)은 `wiring`이 Port에 꽂는다. 지금은 구조만 서 있고 시뮬레이션 로직은 레거시 `tools/`에서 이전해 채워 넣을 차례다.**
